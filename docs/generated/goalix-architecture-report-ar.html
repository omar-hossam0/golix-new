الفكرة العامة

Goalix مش عبارة عن صفحة Dashboard فقط. هو نظام تشغيل لأكاديمية رياضية: عندك مستخدمين بأدوار مختلفة، كل دور له شاشات وصلاحيات، وكل عملية في الواجهة تتحول لطلب API، والـ API يقرأ أو يكتب في PostgreSQL، وأحيانا يستخدم Redis لتسريع القراءة أو تأجيل شغل للـ worker.

أهم جملة تفهم منها النظام: PostgreSQL هو المكان الذي يحفظ الحقيقة النهائية، Redis هو طبقة مساعدة للسرعة والتوزيع والشغل الخلفي، والـ worker هو العامل الذي ينفذ المهام التي لا يجب أن تعطل المستخدم وهو ينتظر الرد.

Frontend

Next.js App Router. فيه مسارات admin/coach/player/parent/auth، وRTK Query لجلب البيانات وإعادة المحاولة عند refresh session أو CSRF.

Backend

Express API. كل module له routes/controller/service/repository. هذا يجعل المنطق مقسم بدل ملف واحد ضخم.

Database

PostgreSQL عبر Knex. أي بيانات مهمة مثل users, sessions, players, matches, attendance, chat, notifications محفوظة هناك.

Redis + Workers

Redis يشغل queues والكاش والrate limits والSocket.IO adapter والlocks. Workers تستهلك queues وتشغل automations.

يعني إيه Job؟ وليه نخزنها؟

كلمة job هنا معناها أمر شغل صغير. تخيل ورقة تعليمات مكتوب عليها: اعمل كذا، ومعاك البيانات دي. الـ API بدل ما ينفذ الشغل الطويل بنفسه وهو المستخدم مستني، يحط ورقة التعليمات دي في queue. بعد كده worker منفصل يقرأ الورقة وينفذها.

مثال بسيط
لما يحصل تعديل حساس في النظام، محتاجين نسجل audit log. بدل ما الطلب يفضل مستني خطوة تسجيل إضافية، الـ API يجهز job اسمها log وفيها بيانات العملية. worker يقرأها ويكتبها في audit_logs.

إذن الجملة “queue بتخزن jobs” معناها: Redis يحتفظ بسجلات صغيرة لكل مهمة: اسمها، بياناتها، حالتها، عدد محاولاتها، وقت تنفيذها، وهل فشلت أو نجحت. هي لا تخزن اللاعبين ولا الأكاديميات كبديل للDB. هي تخزن أوامر تنفيذ مؤقتة أو خلفية.

ليه التخزين مفيد؟
لأنه يسمح للنظام يقول للمستخدم: تم قبول العملية بسرعة، ثم يكمل الشغل غير العاجل في الخلفية. كمان لو عندك ضغط كبير، jobs تصطف بدل ما تكسر السيرفر. ولو worker وقع ورجع، يقدر يكمل من Redis.

Queue في المشروع بالضبط

الملف golx-backend/src/infrastructure/queue.js ينشئ خمس queues: rankings, notifications, payments, ai, audit. كل queue اسمها الحقيقي يبدأ بـ prefix من env، افتراضيا goalix. يعني مثلا audit تصبح goalix-audit.

الـ API يستخدم queue.add(jobName, payload). BullMQ يحفظ job داخل Redis. worker process في production يبدأ من src/workers/run.js، يتصل بPostgreSQL وRedis، ثم يفتح workers لكل queue.

لو Redis غير متاح أثناء audit، الكود لا يضيع الـ audit log: يحاول fallback ويكتب مباشرة في PostgreSQL. لذلك audit queue هنا optimization وفصل حمل، وليست نقطة فشل وحيدة.

أما queues الأخرى فوجودها حاليا يعني أن تصميم النظام جاهز للشغل الخلفي، لكن ليس كل worker مربوط بتنفيذ حقيقي. هذا فرق مهم: architecture جاهزة، لكن بعض integrations لم تكتمل.

Redis: هو بيعمل إيه غير الكاش؟

Redis في Goalix هو shared runtime. يعني حاجة سريعة في الذاكرة تستخدمها كل processes. استخداماته كثيرة: queues، session cache، rate limit counters، socket fanout، distributed locks، وكاشات JSON قصيرة.

Load Balancer / Nginx

الموجود في ops/nginx/nginx.prod.conf حاليا هو Nginx reverse proxy. هو يستقبل المستخدمين على 80/443، يحول HTTP إلى HTTPS، ويرسل المسارات للservice المناسبة.

المساريروح فين؟ملاحظات
/healthAPI healthمسموح على HTTP وHTTPS
/api/v1/Express APIproxy_read_timeout وsend_timeout 120s
/socket.io/Express API Socket.IOيدعم Upgrade/Connection للWebSocket
/uploads/Express APIلأن الملفات protected وليست static
/Next.js frontendأي شيء غير API/socket/uploads
هل هو Load Balancer؟
حاليا لا بالمعنى الأفقي. upstream goalix_api يحتوي server واحد فقط: api:3000. لو أضفت api-1 وapi-2 وapi-3، وقتها Nginx يوزع بينهم افتراضيا round-robin.

Nginx لا يشغل workers ولا يوزع jobs. توزيع الـ jobs يتم لأن كل worker متصل بنفس Redis queue، وBullMQ يعطي كل job لworker واحد يستهلكها.

Performance: اتحسن إزاي؟

الأداء في Goalix ليس feature واحدة. هو مجموعة قرارات تقلل وقت الانتظار وتقلل ضغط DB وتمنع السيرفر من الانهيار تحت الحمل.

فصل worker عن API

في production، api لا يشغل BullMQ workers. هذا يجعل الطلبات السريعة لا تتزاحم مع مهام AI/payment/notifications/ranking.

Connection reuse

Next.js proxy يستخدم http/https agents بـ keepAlive وmaxSockets. Nginx يستخدم upstream keepalive. هذا يقلل تكلفة TCP connections.

Redis short caches

قراءات ساخنة مثل unread count وattendance overview وchat conversations لا تضرب DB كل مرة.

Rate limits

RedisStore للrate limits يجعل الحدود موزعة بين instances، مع fallback memory عند التعطل.

PostgreSQL pool/timeouts

DB_POOL_MAX=10 في production compose، وstatement/lock/idle timeouts تقلل queries العالقة.

Slow logs

SLOW_QUERY_LOG_MS وSLOW_REQUEST_LOG_MS يسمحان بمعرفة المسارات الثقيلة بدل التخمين.

Database: ماشية إزاي؟ وهل كويسة؟

قاعدة البيانات PostgreSQL هي source of truth. الكود يستخدم Knex، ومعظم modules لا تكتب SQL عشوائي داخل controller؛ بل تمر عبر repositories. هذا جيد للفصل والاختبار وتقليل تكرار المنطق.

الدخول في النظام ليس مجرد JWT. فيه cookies، refresh token/session record في DB، optional Redis session cache، CSRF، وMFA للأدوار الإدارية.

  1. المستخدم يرسل username/password من الواجهة.
  2. AuthService يجلب المستخدم ويتأكد من الدور والحالة وكلمة السر.
  3. لو الدور admin أو coach وMFA مطلوب، يرجع mfa_required بدل ما ينشئ session مباشرة.
  4. الـ challenge قد يحفظ في Redis كمفتاح مؤقت، ومعه fallback في PostgreSQL.
  5. بعد OTP صحيح، يتم إنشاء refresh/session record في DB.
  6. Access token وrefresh token يرسلان في cookies آمنة، وRedis يخزن session cache لتسريع الطلبات القادمة.
  7. proxy.ts في Next يفحص accessToken وrole قبل السماح بدخول /admin أو /coach أو /player أو /parent.

Realtime Flow

Realtime في Goalix يستخدم Socket.IO. الفكرة: حفظ البيانات في DB أولا، ثم إرسال event للمتصلين. هذا مهم لأن لو event ضاع، العميل يقدر يعمل REST refresh ويرجع للحقيقة من DB.

  1. المتصفح يفتح socket connection مع cookies أو Bearer token.
  2. Socket server يستخدم authenticateAccessToken للتحقق من المستخدم.
  3. كل user يدخل room باسم user:{userId}.
  4. عند فتح conversation، العميل يرسل chat:join.
  5. الخادم يتأكد من أن المستخدم يملك حق دخول المحادثة.
  6. عند message/notification، البيانات تحفظ في PostgreSQL أولا.
  7. بعد الحفظ، Socket.IO يرسل chat:* أو notification:* للrooms المناسبة.
  8. لو Redis adapter متاح، الحدث ينتقل بين API instances. لو غير متاح، الحدث محلي للinstance فقط.

Protected Files / Uploads

الملفات في المشروع لا ينفع تتساب static public ببساطة، لأن فيها chat attachments أو ملفات تخص لاعب/أكاديمية. لذلك /uploads/* في Express يتحقق من الصلاحية قبل إرسال الملف.

  1. المستخدم يطلب /uploads/path.
  2. authMiddleware يتأكد من access token.
  3. لو path يبدأ بـ chat/، ChatService يتأكد أن المستخدم participant أو مسموح له بالattachment.
  4. لو ملف عادي، storage metadata تبحث في media_files، ثم access policy تحدد هل المستخدم يرى الملف.
  5. لو مسموح، storage adapter يرجع stream/file.
  6. لو مرفوض، النظام يرجع 404 آمن حتى لا يكشف وجود الملف، وقد يسجل audit denied لو حساس.

Production Scaling: ماذا نحتاج لاحقا؟

التصميم يسمح بالتوسع، لكن current compose يشغل API service واحد. للتوسع الحقيقي تضيف أكثر من API instance، أكثر من worker، وتضع PgBouncer أمام PostgreSQL.

حدود وملاحظات صريحة

  • الـ PDF الحالي لا يحتوي Mermaid code. الأكواد موجودة فقط في ملف Markdown الملحق.
  • Audit queue فعالة فعلا. أما AI/ranking/payment/external notifications workers فهي في أغلبها stubs/TODO حاليا.
  • Nginx الحالي reverse proxy وليس load balancer أفقي لأن upstream فيه server واحد.
  • Redis مهم جدا للتشغيل الكامل، لكنه ليس مصدر الحقيقة للبيانات الأساسية.
  • قاعدة البيانات منظمة ومناسبة كبداية قوية، لكن إثبات تحمل أرقام كبيرة يحتاج load testing ومراقبة حقيقية.
  • يوجد تكرار في nginx config لنفس location / ثلاث مرات. السلوك غالبا ثابت، لكن الأفضل تنظيفه.
